1. 技术概述
2. 关键实现细节

### 1. 知识图谱提取工具 (`KnowledgeGraphTool`)

#### a) 代码片段

```python
# src/crew2/tools/custom_tools.py

from crewai_tools import BaseTool
from serpapi import GoogleSearch
from dotenv import load_dotenv
import os

# 加载环境变量
load_dotenv()
serper_api_key = os.getenv('SERPER_API_KEY')

# 知识图谱工具
class KnowledgeGraphTool(BaseTool):
    name: str = "知识图谱提取器"
    description: str = (
        "该工具从Google知识图谱中提取特定信息。 "
        "如果没有可用的知识图谱数据，则回退到其他工具。"
    )

    def _run(self, query: str) -> dict:
        params = {
            'engine': 'google',
            'q': query,
            'api_key': serper_api_key
        }
        search = GoogleSearch(params)
        results = search.get_dict()
        knowledge_graph = results.get('knowledge_graph', None)
        return {"knowledge_graph": knowledge_graph} if knowledge_graph else {"error": "未找到知识图谱数据。"}
```

#### b) 实现步骤说明

- **目的**：
  - 从Google知识图谱中提取与查询相关的结构化数据。
  
- **主要函数**：
  - `_run(self, query: str) -> dict`：执行查询并返回知识图谱数据或错误信息。
  
- **关键变量**：
  - `serper_api_key`：用于SerpAPI的认证密钥。
  - `params`：查询参数，包括引擎类型、查询内容和API密钥。
  
- **控制流**：
  1. 构建查询参数。
  2. 使用SerpAPI的`GoogleSearch`执行搜索。
  3. 获取搜索结果，并尝试提取`knowledge_graph`部分。
  4. 返回知识图谱数据或错误信息。
  
- **错误处理**：
  - 如果未找到`knowledge_graph`数据，返回包含错误信息的字典。
  
- **性能考虑**：
  - 优化API调用，通过只请求必要的数据字段减少响应时间。

#### c) 技术决策及理由

- **选择SerpAPI**：
  - SerpAPI提供了可靠的Google搜索接口，支持结构化数据的提取，如知识图谱。
  
- **回退机制**：
  - 如果知识图谱数据不可用，允许系统回退到通用搜索工具，确保数据获取的连续性。
  
- **替代方案**：
  - 考虑使用Google官方的API，但由于限制和成本，选择SerpAPI更为合适。
  
- **权衡**：
  - SerpAPI虽然是付费服务，但提供了稳定且详尽的搜索结果，值得投资。

#### d) 与外部服务的集成点

- **认证**：
  - 通过环境变量`SERPER_API_KEY`进行SerpAPI的认证。
  
- **数据格式化**：
  - SerpAPI返回JSON格式的数据，工具需要解析并提取`knowledge_graph`部分。
  
- **响应处理**：
  - 检查`knowledge_graph`是否存在，并根据结果返回相应的数据或错误信息。

#### e) 优化技术

- **缓存机制**：
  - 可以引入缓存策略，避免对相同查询的重复API调用，提升性能。
  
- **并发请求**：
  - 对多个查询进行并发处理，减少总体等待时间。

#### f) 配置参数及其影响

- **`SERPER_API_KEY`**：
  - 影响工具是否能够成功调用SerpAPI并获取数据。
  
- **查询参数**：
  - 可以根据需求调整搜索引擎类型、查询内容等，影响数据获取的精确性和范围。

### 2. 通用Google搜索工具 (`OrganicSearchTool`)

#### a) 代码片段

```python
# src/crew2/tools/custom_tools.py

from crewai_tools import BaseTool
from serpapi import GoogleSearch
from dotenv import load_dotenv
import os

# 加载环境变量
load_dotenv()
serper_api_key = os.getenv('SERPER_API_KEY')

# 通用Google搜索工具
class OrganicSearchTool(BaseTool):
    name: str = "通用Google搜索"
    description: str = (
        "执行通用的有机Google搜索并检索有机结果。 "
        "如果没有知识图谱数据可用，请使用此工具。"
    )

    def _run(self, query: str) -> dict:
        params = {
            "engine": "google",
            "q": query,
            "api_key": serper_api_key
        }
        search = GoogleSearch(params)
        results = search.get_dict()
        organic_results = results.get("organic_results", None)

        if organic_results:
            # 返回前6个结果
            top_results = organic_results[:6]
            return {"results": top_results}
        else:
            return {"error": "未找到通用搜索结果。"}
```

#### b) 实现步骤说明

- **目的**：
  - 执行通用的Google有机搜索，获取相关的搜索结果。
  
- **主要函数**：
  - `_run(self, query: str) -> dict`：执行搜索并返回前6个有机结果或错误信息。
  
- **关键变量**：
  - `serper_api_key`：用于SerpAPI的认证密钥。
  - `params`：查询参数，包括引擎类型、查询内容和API密钥。
  
- **控制流**：
  1. 构建查询参数。
  2. 使用SerpAPI的`GoogleSearch`执行搜索。
  3. 获取搜索结果，并提取`organic_results`部分。
  4. 返回前6个有机搜索结果或错误信息。
  
- **错误处理**：
  - 如果未找到`organic_results`，返回包含错误信息的字典。
  
- **性能考虑**：
  - 限制返回结果数量，减少数据处理和传输时间。

#### c) 技术决策及理由

- **限制返回结果数量**：
  - 仅返回前6个结果，确保结果的相关性和处理效率。
  
- **选择SerpAPI**：
  - 与知识图谱工具保持一致，方便统一管理API调用。
  
- **替代方案**：
  - 使用其他搜索API，如Bing Search API，但SerpAPI在数据丰富性和稳定性方面更具优势。
  
- **权衡**：
  - 虽然限制了返回结果数量，但能够提升处理速度，同时保留足够的信息供后续处理。

#### d) 与外部服务的集成点

- **认证**：
  - 通过环境变量`SERPER_API_KEY`进行SerpAPI的认证。
  
- **数据格式化**：
  - SerpAPI返回JSON格式的数据，工具需要解析并提取`organic_results`部分。
  
- **响应处理**：
  - 检查`organic_results`是否存在，并根据结果返回相应的数据或错误信息。

#### e) 优化技术

- **分页处理**：
  - 根据需要处理更多搜索结果，支持分页查询。
  
- **结果过滤**：
  - 对结果进行过滤和排序，提升结果的相关性和质量。

#### f) 配置参数及其影响

- **`SERPER_API_KEY`**：
  - 影响工具是否能够成功调用SerpAPI并获取数据。
  
- **查询参数**：
  - 调整搜索引擎类型、查询内容和结果数量，影响搜索结果的相关性和范围。

### 3. 研究员代理 (`researcher`)

@agent
def researcher(self) -> Agent:

### 4. 任务定义（研究任务和报告任务）

#### a) 代码片段

```yaml
# src/crew2/config/tasks.yaml

research_task:
  description: >
    进行对{topic}的深入调查，主要使用Google知识图谱作为结构化数据的来源。 
    如果没有知识图谱信息，改为使用通用Google搜索工具，检索有机搜索结果的前2个URL，并使用抓取工具提取相关网页内容。
    优先保证发现的准确性、相关性和清晰性。
  expected_output: >
    一份详细且准确的研究报告，总结结构化的知识图谱数据（如果可用）。如果不可用，则提供从通用搜索中收集的见解，确保所有信息与{topic}相关。如果通用搜索未被授权，或未获取到信息，则仅报告该情况。
  agent: researcher

reporting_task:
  description: >
    如果生成了任何报告，审核关于{topic}的详细研究报告，并将其转化为结构良好且简洁的报告。
    确保报告简洁明了，包含构建{topic}档案所需的所有相关信息。
  expected_output: >
    如果有任何知识图谱或通用搜索的报告，提供{topic}的综合档案。确保报告围绕关键信息进行结构化，并足够详细以提供对主题的全面理解。报告应包括但不限于以下部分：
    - 基本信息：名称、类型（个人、实体、组织）及相关类别或行业。
    - 背景：历史背景、起源或成立详情。
    - 关键人物：与主题相关的重要人物，如创始人、关键高管或知名人士，及其简要简介。
    - 位置：总部或主要位置（如适用）。
    - 财务信息：相关财务数据，如股票信息、融资情况或经济影响（如适用）。
    - 联系信息：客户服务详情、官方联系链接或其他相关联系信息。
    - 在线存在感：官方网站链接、社交媒体简介及其他在线资源。
    - 其他相关信息：任何其他有助于全面理解主题的细节。
    如果未生成报告，仅报告该情况。
  agent: reporter
```

#### b) 实现步骤说明

- **目的**：
  - 定义研究任务和报告任务的描述、预期输出及关联代理。
  
- **任务定义**：
  - **`research_task`**：
    - 指定使用研究员代理，主要任务是收集与主题相关的结构化或非结构化数据。
  - **`reporting_task`**：
    - 指定使用报告员代理，主要任务是将研究结果整理成结构化的报告。
  
- **关键变量**：
  - `description`：任务的详细描述，指导代理的行为和目标。
  - `expected_output`：任务期望的输出结果，定义了结果的格式和内容。
  - `agent`：关联执行任务的代理角色。
  
- **控制流**：
  1. 研究任务执行，收集相关数据。
  2. 如果有研究报告生成，进入报告任务，整理和优化报告内容。
  
- **错误处理**：
  - 如果研究任务未能获取到数据，记录并报告该情况。
  - 如果报告任务未能生成报告，记录并报告该情况。
  
- **性能考虑**：
  - 定义清晰的预期输出，有助于任务快速判断是否完成，提升整体执行效率。

#### c) 技术决策及理由

- **任务拆分**：
  - 将数据收集和报告整理分为两个独立的任务，职责清晰，便于维护和扩展。
  
- **灵活的任务描述**：
  - 使用占位符`{topic}`，提高任务定义的可复用性和灵活性。
  
- **预期输出详细定义**：
  - 明确任务的期望结果，有助于代理准确执行任务并生成符合需求的输出。
  
- **关联不同的代理**：
  - 不同任务关联不同的代理，充分利用各自的优势，提升整体系统的效率和效果。
